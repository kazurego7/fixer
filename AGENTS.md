# AGENTS

## UI不具合の再現確認

- UI不具合では、再現検証と原因分析を憶測で終わらせず、まず `agent-browser` を使って事実ベースで確認する。
- 再現確認では、同じ条件を繰り返せるように前提条件と操作条件を明確にする。
- 必要に応じて観測値を取り、印象ではなく確認可能な事実で判断する。
- 再現できた場合は、再現条件、観測結果、期待との差分を残す。
- 修正時は、再現した条件を E2Eテストに落とし込む。
- E2Eテストでは、修正前に失敗し、修正後に成功する `red -> green` を確認する。
- 修正前後の見た目に差が出る場合は、`before` / `after` のスクリーンショットを残し、人間が比較確認できるようにする。

## 機能追加時のUI確認

- 新機能や UI変更を入れたときは、必要に応じて状態別の E2Eテストを追加し、見た目確認用のスクリーンショット成果物を残す。
- 動作確認では、テストの pass/fail だけで終わらず、`playwright-report` や `test-results` の画像を見て見かけを確認する。
- 確認用の UI を本体に埋め込む前に、E2E のモックや専用 spec で状態を再現できないかを優先して検討する。
- 色、余白、表示順、導線のような見た目に関わる変更では、スクリーンショットで変更前後や状態差分を比較できるようにする。
- スクリーンショットは回帰の自動判定よりも人間の確認を助ける成果物として扱い、必要に応じて `before` / `after` を並べて確認できる形で残す。

## ログ取得方法

### サーバーログ（推奨）
- エンドポイント: `GET /api/logs`
- 例:
  - `curl -sS 'http://localhost:3000/api/logs?limit=50'`
  - `curl -sS 'http://localhost:3000/api/logs?limit=200&level=error'`

### 動作確認用
- ヘルスチェック:
  - `curl -sS 'http://localhost:3000/api/health'`
- スレッド一覧確認:
  - `curl -sS 'http://localhost:3000/api/threads?repoFullName=OWNER%2FREPO'`

### トラブルシュート時の確認順
1. `GET /api/health` でサーバー稼働確認
2. `GET /api/logs?limit=100` で直近ログ確認
3. 必要に応じて `level=error` で絞り込み
4. 送信不具合時は、送信直後のログを時刻付きで確認
